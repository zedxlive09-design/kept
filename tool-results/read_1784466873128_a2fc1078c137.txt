    25‚Üí    25‚Üí## 1. Product Overview
    26‚Üí    26‚Üí
    27‚Üí    27‚Üí**One-liner:** Kept is a personal purchase vault ‚Äî receipts, warranties, subscriptions, and bills in one place, with AI-assisted data entry and reminders before you lose money to a missed warranty claim or a forgotten renewal.
    28‚Üí    28‚Üí
    29‚Üí    29‚Üí**Shape of the product now:** a public marketing landing page (¬ß8) that explains the pitch and sells the sign-up, sitting in front of the authenticated app (dashboard, items, subscriptions, reminders, settings). Today it ships as a website. The same codebase is built, from the first line of code, so it can be wrapped as an Android APK via Capacitor later without a rewrite ‚Äî see ¬ß3.
    30‚Üí    30‚Üí
    31‚Üí    31‚Üí**Target users:** individuals and families managing personal purchases and subscriptions; secondarily freelancers/solo founders tracking tool subscriptions and hardware warranties.
    32‚Üí    32‚Üí
    33‚Üí    33‚Üí**Core value loop:** land on the marketing page ‚Üí sign up ‚Üí snap a photo ‚Üí AI fills in the form ‚Üí user confirms ‚Üí item is tracked ‚Üí user gets reminded before it expires or renews.

    55‚Üí    55‚Üí## 3. Why This Architecture: Website Today, APK Tomorrow
    56‚Üí    56‚Üí
    57‚Üí    57‚Üí**Why this module exists:** this is the section that makes the Capacitor requirement real instead of aspirational. Bolting Capacitor onto an app that was built with Server Actions and dynamic server routes means rebuilding the data layer later. Building it in from day one costs almost nothing extra now and saves a rewrite later.
    58‚Üí    58‚Üí
    59‚Üí    59‚Üí**What static export changes, concretely:**
    60‚Üí    60‚Üí
    61‚Üí    61‚Üí| Normal Next.js (what the old spec assumed) | This spec (static-export safe) | Why |
    62‚Üí    62‚Üí|---|---|---|
    63‚Üí    63‚Üí| Server Actions for writes | Supabase client SDK called directly from Client Components | Server Actions need a live Node server at request time. A statically exported bundle running inside a Capacitor WebView has no server behind it at all. |
    64‚Üí    64‚Üí| Route Handler (`/api/extract-receipt`) for the Gemini call | Supabase **Edge Function** (`supabase.functions.invoke()`) | Same reasoning ‚Äî the Edge Function is a separately-hosted HTTPS endpoint that works identically whether the UI calling it is a browser tab on Vercel or a Capacitor WebView on a phone with no Vercel deployment in the picture at all. |
    65‚Üí    65‚Üí| `/items/[id]` dynamic route segment | `/items/detail?id=...` ‚Äî a **static page that reads `id` from `useSearchParams()`** | Static export requires every route to be fully known at build time (`generateStaticParams`). Item IDs are created by users at runtime ‚Äî they can't be pre-generated. A query param sidesteps this entirely since `/items/detail` itself is one static page. |
    66‚Üí    66‚Üí| Next.js Middleware for auth-gating routes | A client-side `<AuthGuard>` wrapper that checks `supabase.auth.getSession()` on mount and redirects if absent | Middleware requires an edge/server runtime that doesn't exist once the app is bundled into Capacitor. |
    67‚Üí    67‚Üí| `next/image` optimization | `images: { unoptimized: true }` in `next.config.js`, plain `<img>` | The optimizer is a server feature; it silently can't run in a static/Capacitor context. |
    68‚Üí    68‚Üí
    69‚Üí    69‚Üí**The payoff:** the exact same `out/` build directory that gets deployed to Vercel today becomes the `webDir` in `capacitor.config.ts` when you're ready to build the APK. One codebase, two shells, zero duplicated logic. Details on that later step are in ¬ß16 ‚Äî don't build it now, just don't box yourself out of it.

   110‚Üí   110‚Üí## 5. Database Schema
   111‚Üí   111‚Üí
   112‚Üí   112‚Üí**Why this module exists:** this is the shared source of truth for every screen in the app ‚Äî the dashboard, the list, the reminders, the AI review form all read and write the same two tables. Getting the shape right once here avoids three different half-compatible data models emerging across features later.
   113‚Üí   113‚Üí
   114‚Üí   114‚Üí**Why one `items` table instead of three:** a purchase, a subscription, and a bill are the same thing to the user ‚Äî "something I need to track before it costs me money." Splitting them into `purchases`/`subscriptions`/`bills` tables would mean three near-identical schemas and a `UNION` every time the dashboard or search needs to show all of them together. One table with a `type` discriminator and nullable type-specific columns keeps the common 90% (name, amount, status, reminders) in one place and isolates the 10% that actually differs.
   115‚Üí   115‚Üí
   116‚Üí   116‚Üí```sql
   117‚Üí   117‚Üí-- ============================================================
   118‚Üí   118‚Üí-- ENUMS
   119‚Üí   119‚Üí-- ============================================================
   120‚Üí   120‚Üícreate type item_type as enum ('purchase', 'subscription', 'bill');
   121‚Üí   121‚Üícreate type item_status as enum ('active', 'expiring_soon', 'expired', 'cancelled');
   122‚Üí   122‚Üícreate type reminder_kind as enum ('warranty_expiry', 'subscription_renewal', 'bill_due');
   123‚Üí   123‚Üí
   124‚Üí   124‚Üí-- ============================================================
   125‚Üí   125‚Üí-- PROFILES ‚Äî why: timezone-correct reminders need somewhere to
   126‚Üí   126‚Üí-- store each user's IANA timezone, and Settings needs an email
   127‚Üí   127‚Üí-- opt-in toggle. Auto-created on signup so it's never missing.
   128‚Üí   128‚Üí-- ============================================================
   129‚Üí   129‚Üícreate table public.profiles (
   130‚Üí   130‚Üí  id uuid primary key references auth.users(id) on delete cascade,
   131‚Üí   131‚Üí  display_name text,
   132‚Üí   132‚Üí  timezone text not null default 'UTC',
   133‚Üí   133‚Üí  email_reminders_enabled boolean not null default true,
   134‚Üí   134‚Üí  created_at timestamptz not null default now()
   135‚Üí   135‚Üí);
   136‚Üí   136‚Üí
   137‚Üí   137‚Üícreate function public.handle_new_user()
   138‚Üí   138‚Üíreturns trigger
   139‚Üí   139‚Üílanguage plpgsql
   140‚Üí   140‚Üísecurity definer set search_path = public
   141‚Üí   141‚Üías $$
   142‚Üí   142‚Üíbegin
   143‚Üí   143‚Üí  insert into public.profiles (id, display_name)
   144‚Üí   144‚Üí  values (new.id, new.raw_user_meta_data->>'full_name');
   145‚Üí   145‚Üí  return new;
   146‚Üí   146‚Üíend;
   147‚Üí   147‚Üí$$;
   148‚Üí   148‚Üí
   149‚Üí   149‚Üícreate trigger on_auth_user_created
   150‚Üí   150‚Üí  after insert on auth.users
   151‚Üí   151‚Üí  for each row execute function public.handle_new_user();
   152‚Üí   152‚Üí
   153‚Üí   153‚Üícreate function public.set_updated_at()
   154‚Üí   154‚Üíreturns trigger language plpgsql as $$
   155‚Üí   155‚Üíbegin
   156‚Üí   156‚Üí  new.updated_at = now();
   157‚Üí   157‚Üí  return new;
   158‚Üí   158‚Üíend;
   159‚Üí   159‚Üí$$;
   160‚Üí   160‚Üí
   161‚Üí   161‚Üí-- ============================================================
   162‚Üí   162‚Üí-- ITEMS ‚Äî the core table. See rationale above.
   163‚Üí   163‚Üí-- ============================================================
   164‚Üí   164‚Üícreate table public.items (
   165‚Üí   165‚Üí  id uuid primary key default gen_random_uuid(),

   167‚Üí   167‚Üí  type item_type not null,
   168‚Üí   168‚Üí  status item_status not null default 'active',
   169‚Üí   169‚Üí
   170‚Üí   170‚Üí  name text not null,
   171‚Üí   171‚Üí  merchant text,
   172‚Üí   172‚Üí  category text not null default 'other',   -- free text + client-side presets, no categories table for MVP

   257‚Üí   257‚Üí-- The daily cron Edge Function (¬ß10) uses the SERVICE ROLE key
   258‚Üí   258‚Üí-- and bypasses RLS by design ‚Äî the one place allowed to touch
   259‚Üí   259‚Üí-- every user's rows.
   260‚Üí   260‚Üí```

   264‚Üí   264‚Üí## 6. Storage Architecture
   265‚Üí   265‚Üí
   266‚Üí   266‚Üí**Why this module exists:** receipt images are the one piece of user data that isn't a database row, and they're the thing most likely to blow through the free storage tier if handled carelessly.
   267‚Üí   267‚Üí
   268‚Üí   268‚Üí- **Bucket:** `receipts` ‚Äî private.
   269‚Üí   269‚Üí- **Path:** `{user_id}/{item_id}/{timestamp}-{filename}`. Generate the item's UUID client-side (`crypto.randomUUID()`) before upload, so the storage path and eventual DB row share the same ID.
   270‚Üí   270‚Üí
   271‚Üí   271‚Üí```sql
   272‚Üí   272‚Üíinsert into storage.buckets (id, name, public) values ('receipts', 'receipts', false);
   273‚Üí   273‚Üí
   274‚Üí   274‚Üícreate policy "receipts_select_own"
   275‚Üí   275‚Üíon storage.objects for select
   276‚Üí   276‚Üíusing (bucket_id = 'receipts' and (storage.foldername(name))[1] = (select auth.uid())::text);
   277‚Üí   277‚Üí
   278‚Üí   278‚Üícreate policy "receipts_insert_own"
   279‚Üí   279‚Üíon storage.objects for insert
   280‚Üí   280‚Üíwith check (bucket_id = 'receipts' and (storage.foldername(name))[1] = (select auth.uid())::text);
   281‚Üí   281‚Üí
   282‚Üí   282‚Üícreate policy "receipts_delete_own"
   283‚Üí   283‚Üíon storage.objects for delete
   284‚Üí   284‚Üíusing (bucket_id = 'receipts' and (storage.foldername(name))[1] = (select auth.uid())::text);
   285‚Üí   285‚Üí```
   286‚Üí   286‚Üí
   287‚Üí   287‚Üí**Two gotchas to build around from day one, not patch in later:**
   288‚Üí   288‚Üí1. **HEIC images.** iPhones default to HEIC, which most browsers won't render inline. Convert to JPEG client-side (`heic2any`) before upload.
   289‚Üí   289‚Üí2. **Storage budget.** Free tier is 1 GB. Compress client-side (`browser-image-compression`, ~1600px max width, ~80% JPEG quality) before upload ‚Äî a raw phone photo can be 4‚Äì8 MB; compressed, it's closer to 200‚Äì400 KB, roughly a 15‚Äì20x runway extension for the same number of receipts.

   293‚Üí   293‚Üí## 7. AI Receipt Extraction Pipeline
   294‚Üí   294‚Üí
   295‚Üí   295‚Üí**Why this module exists ‚Äî and why it moved off Next.js:** this is the one piece of the app that touches a secret API key, so it can never run in the browser. In the original spec it lived in a Next.js Route Handler; here it's a **Supabase Edge Function** instead, purely because of ¬ß3 ‚Äî a Route Handler only exists while a Next.js server is running, and once this ships as a Capacitor APK with no such server behind it, that endpoint would simply vanish. An Edge Function is a standalone HTTPS endpoint from day one, so the AI pipeline works identically whether it's called from a browser tab or a phone app.
   296‚Üí   296‚Üí
   297‚Üí   297‚Üí**Flow:** user photographs/uploads a receipt ‚Üí image uploaded to Storage ‚Üí client calls the Edge Function with the storage path ‚Üí function fetches the image, calls Gemini with a JSON schema ‚Üí structured fields return ‚Üí **user reviews and edits before saving ‚Äî nothing is auto-saved from AI output.**
   298‚Üí   298‚Üí
   299‚Üí   299‚Üí**The rule this pipeline must never break** (carried over from the original product discovery): *never let the AI infer or estimate a warranty period.* Most receipts don't print one. `warranty_months` stays `null` unless it's explicitly printed on the document ‚Äî the UI always renders that field as empty-and-editable rather than AI-guessed, so a wrong guess never quietly becomes a saved fact.
   300‚Üí   300‚Üí
   301‚Üí   301‚Üí```typescript
   302‚Üí   302‚Üí// supabase/functions/extract-receipt/index.ts
   303‚Üí   303‚Üí// Deno runtime. Invoked via supabase.functions.invoke('extract-receipt', {...})
   304‚Üí   304‚Üí// which automatically forwards the caller's JWT ‚Äî no separate auth wiring needed.
   305‚Üí   305‚Üí
   306‚Üí   306‚Üíconst EXTRACTION_PROMPT = `You are extracting structured data from a photo of a receipt,
   307‚Üí   307‚Üíinvoice, or subscription confirmation email screenshot.
   308‚Üí   308‚ÜíOnly return fields you can actually read from the image.
   309‚Üí   309‚ÜíDo NOT guess or estimate a warranty period. Only set warranty_months if a warranty
   310‚Üí   310‚Üíduration is explicitly printed on the document. Otherwise return null for it.`;
   311‚Üí   311‚Üí
   312‚Üí   312‚Üíconst responseSchema = {
   313‚Üí   313‚Üí  type: "OBJECT",
   314‚Üí   314‚Üí  properties: {
   315‚Üí   315‚Üí    name:               { type: "STRING" },
   316‚Üí   316‚Üí    merchant:           { type: "STRING" },
   317‚Üí   317‚Üí    amount:             { type: "NUMBER" },
   318‚Üí   318‚Üí    currency:           { type: "STRING" },
   319‚Üí   319‚Üí    purchase_date:      { type: "STRING", description: "YYYY-MM-DD" },
   320‚Üí   320‚Üí    suggested_category: { type: "STRING" },
   321‚Üí   321‚Üí    suggested_type:     { type: "STRING", enum: ["purchase", "subscription", "bill"] },
   322‚Üí   322‚Üí    warranty_months:    { type: "INTEGER", nullable: true },
   323‚Üí   323‚Üí    confidence_note:    { type: "STRING", description: "Anything illegible or ambiguous" }
   324‚Üí   324‚Üí  },
   325‚Üí   325‚Üí  required: ["name", "merchant", "amount"]
   326‚Üí   326‚Üí};
   327‚Üí   327‚Üí
   328‚Üí   328‚ÜíDeno.serve(async (req) => {
   329‚Üí   329‚Üí  // 1. Extract + verify the caller from the forwarded Authorization header
   330‚Üí   330‚Üí  // 2. Rate-limit check ‚Äî see below
   331‚Üí   331‚Üí  // 3. Fetch the image bytes from Storage using the path the client sends
   332‚Üí   332‚Üí  // 4. Call Gemini:
   333‚Üí   333‚Üí  const res = await fetch(
   334‚Üí   334‚Üí    `https://generativelanguage.googleapis.com/v1beta/models/gemini-3.5-flash:generateContent?key=${Deno.env.get("GEMINI_API_KEY")}`,
   335‚Üí   335‚Üí    {
   336‚Üí   336‚Üí      method: "POST",
   337‚Üí   337‚Üí      headers: { "Content-Type": "application/json" },
   338‚Üí   338‚Üí      body: JSON.stringify({
   339‚Üí   339‚Üí        contents: [{ parts: [{ text: EXTRACTION_PROMPT }, { inline_data: { mime_type: "image/jpeg", data: base64Image } }] }],
   340‚Üí   340‚Üí        generationConfig: { responseMimeType: "application/json", responseSchema }
   341‚Üí   341‚Üí      })
   342‚Üí   342‚Üí    }
   343‚Üí   343‚Üí  );
   344‚Üí   344‚Üí  // 5. Parse and return the JSON to the client for the review form.
   345‚Üí   345‚Üí  // 6. On any failure ‚Äî return a clear error and let the client fall through
   346‚Üí   346‚Üí  //    to the manual-entry form. AI extraction accelerates the form; it's
   347‚Üí   347‚Üí  //    never a hard dependency for saving an item.
   348‚Üí   348‚Üí});
   349‚Üí   349‚Üí```
   350‚Üí   350‚Üí
   351‚Üí   351‚ÜíSecrets for an Edge Function are set via the Supabase CLI (`supabase secrets set GEMINI_API_KEY=...`), not a `.env.local` file ‚Äî that file only ever holds the *public* Supabase URL/anon key on the Next.js side.
   352‚Üí   352‚Üí
   353‚Üí   353‚Üí**Rate limiting:** before calling Gemini, count today's rows where `user_id = caller AND is_ai_extracted = true AND created_at::date = current_date`. Cap at ~20/day per user ‚Äî generous for personal use, no Redis/Upstash required.

   405‚Üí   405‚Üí## 10. Reminders & Notifications System
   406‚Üí   406‚Üí
   407‚Üí   407‚Üí**Why this module exists:** the entire value proposition ("never lose a warranty or a subscription again") lives in this module. Everything else in the app is bookkeeping; this is the part that actually saves the user money without them doing anything.
   408‚Üí   408‚Üí
   409‚Üí   409‚Üí**Generation (application code, run right after an item is created or updated):**
   410‚Üí   410‚Üí
   411‚Üí   411‚Üí```typescript
   412‚Üí   412‚Üí// lib/reminders.ts
   413‚Üí   413‚Üíconst WARRANTY_WINDOWS_DAYS = [30, 7];
   414‚Üí   414‚Üíconst RENEWAL_WINDOW_DAYS = 3;
   415‚Üí   415‚Üí
   416‚Üí   416‚Üíasync function syncRemindersForItem(supabase: SupabaseClient, item: Item) {
   417‚Üí   417‚Üí  // Clear only *unsent* future reminders for this item, then regenerate.
   418‚Üí   418‚Üí  // Sent reminders are never touched ‚Äî they're the audit trail.
   419‚Üí   419‚Üí  await supabase.from('reminders').delete().eq('item_id', item.id).eq('sent', false);
   420‚Üí   420‚Üí
   421‚Üí   421‚Üí  const rows: ReminderInsert[] = [];
   422‚Üí   422‚Üí
   423‚Üí   423‚Üí  if (item.type === 'purchase' && item.warranty_expiry) {
   424‚Üí   424‚Üí    for (const days of WARRANTY_WINDOWS_DAYS) {
   425‚Üí   425‚Üí      rows.push({ item_id: item.id, user_id: item.user_id, kind: 'warranty_expiry',
   426‚Üí   426‚Üí                  remind_on: subDays(item.warranty_expiry, days) });
   427‚Üí   427‚Üí    }
   428‚Üí   428‚Üí  }
   429‚Üí   429‚Üí  if ((item.type === 'subscription' || item.type === 'bill') && item.next_billing_date) {
   430‚Üí   430‚Üí    rows.push({ item_id: item.id, user_id: item.user_id,
   431‚Üí   431‚Üí                kind: item.type === 'subscription' ? 'subscription_renewal' : 'bill_due',
   432‚Üí   432‚Üí                remind_on: subDays(item.next_billing_date, RENEWAL_WINDOW_DAYS) });
   433‚Üí   433‚Üí  }
   434‚Üí   434‚Üí
   435‚Üí   435‚Üí  const future = rows.filter(r => r.remind_on >= todayISO());
   436‚Üí   436‚Üí  if (future.length) await supabase.from('reminders').upsert(future, { onConflict: 'item_id,kind,remind_on' });
   437‚Üí   437‚Üí}
   438‚Üí   438‚Üí```
   439‚Üí   439‚Üí
   440‚Üí   440‚Üí**Daily cron (Edge Function, triggered by `pg_cron`):**
   441‚Üí   441‚Üí
   442‚Üí   442‚Üí```sql
   443‚Üí   443‚Üíselect cron.schedule(
   444‚Üí   444‚Üí  'send-daily-reminders', '0 7 * * *',
   445‚Üí   445‚Üí  $$ select net.http_post(
   446‚Üí   446‚Üí       url := 'https://<project-ref>.supabase.co/functions/v1/send-reminders',
   447‚Üí   447‚Üí       headers := jsonb_build_object('Authorization', 'Bearer ' || '<service-role-key-from-vault>')
   448‚Üí   448‚Üí     ); $$
   449‚Üí   449‚Üí);
   450‚Üí   450‚Üí```
   451‚Üí   451‚Üí
   452‚Üí   452‚ÜíEdge Function logic:
   453‚Üí   453‚Üí1. Query due, unsent reminders **against each user's local date**, not server UTC: `remind_on <= (now() at time zone p.timezone)::date`, joined to `profiles`. A naive UTC-only comparison sends reminders a day early or late for anyone outside UTC.
   454‚Üí   454‚Üí2. Group by user, send **one email per user** per day listing all due items ‚Äî not one email per reminder.
   455‚Üí   455‚Üí3. Mark sent rows `sent = true, sent_at = now()`.
   456‚Üí   456‚Üí4. **Subscription rollover:** for any `subscription`/`bill` where `next_billing_date < today`, advance it by one `billing_cycle` interval and re-run the sync logic so the *next* cycle's reminder gets created. Skip this and a subscription's reminders silently stop after the first renewal.
   457‚Üí   457‚Üí
   458‚Üí   458‚Üí> ‚öÝÔ∏è **Free-tier gotcha:** Supabase free projects can pause after a period of inactivity. A once-daily cron hit generally keeps a project active, but if reminders silently stop firing, check the project isn't paused before debugging application logic.

   462‚Üí   462‚Üí## 11. Auth & Security
   463‚Üí   463‚Üí
   464‚Üí   464‚Üí**Why this module exists:** RLS (¬ß5) is the actual security boundary ‚Äî this module is about not accidentally undermining it.
   465‚Üí   465‚Üí
   466‚Üí   466‚Üí- **Supabase Auth**, email/password for MVP. Google OAuth is a cheap add later if signup friction becomes a problem.
   467‚Üí   467‚Üí- **Client-side route guard, not middleware** (¬ß3): a shared `<AuthGuard>` in the layout for every page under the authenticated section checks `supabase.auth.getSession()` on mount, subscribes to `onAuthStateChange`, and redirects to `/login` if there's no session.
   468‚Üí   468‚Üí- **RLS is the only authorization layer** ‚Äî no parallel permission checks in application code to keep in sync.
   469‚Üí   469‚Üí- **Secrets never reach the client:** `GEMINI_API_KEY`, `RESEND_API_KEY`, and the Supabase **service role** key live only in Edge Function secrets, never in `NEXT_PUBLIC_*` variables. Only the `anon` key and URL are public ‚Äî safe by design because RLS, not key secrecy, is what protects the data.
   470‚Üí   470‚Üí- **The extraction endpoint is the one abuse surface** (it costs real money per call) ‚Äî the ¬ß7 rate limit and auth check are both non-optional.

   474‚Üí   474‚Üí## 12. Query Optimization Guidelines
   475‚Üí   475‚Üí
   476‚Üí   476‚Üí**Why this module exists:** none of this matters at 20 test rows. It all matters at 2,000 real ones, and retrofitting it after the fact means migrating a live user's data ‚Äî cheaper to build it right the first time.
   477‚Üí   477‚Üí
   478‚Üí   478‚Üí| Index | Why |
   479‚Üí   479‚Üí|---|---|
   480‚Üí   480‚Üí| `(user_id)` on items/reminders | Every RLS policy filters on this ‚Äî the single most-hit predicate in the app |
   481‚Üí   481‚Üí| `(user_id, status)`, `(user_id, type)` | Dashboard and list filters combine these; composite index avoids a second lookup |
   482‚Üí   482‚Üí| Partial index on `warranty_expiry where type='purchase'` | The cron scan only ever touches purchase rows for this field |
   483‚Üí   483‚Üí| `gin_trgm_ops` on `name`/`merchant` | Powers the search bar (¬ß9.3) without a separate search service |
   484‚Üí   484‚Üí| Partial index on `reminders.remind_on where sent=false` | The daily cron's core query ‚Äî sent rows are permanently irrelevant to it |
   485‚Üí   485‚Üí
   486‚Üí   486‚Üí**RLS performance:** every policy wraps `auth.uid()` as `(select auth.uid())` (already done in ¬ß5) so Postgres caches it once per statement instead of re-evaluating per row.
   487‚Üí   487‚Üí
   488‚Üí   488‚Üí**Keyset pagination, not `OFFSET`:**
   489‚Üí   489‚Üí```typescript
   490‚Üí   490‚Üí// first page
   491‚Üí   491‚Üísupabase.from('items').select('id,name,merchant,amount,status,created_at')
   492‚Üí   492‚Üí  .eq('user_id', userId)
   493‚Üí   493‚Üí  .order('created_at', { ascending: false }).order('id', { ascending: false })
   494‚Üí   494‚Üí  .limit(20);
   495‚Üí   495‚Üí
   496‚Üí   496‚Üí// next page ‚Äî cursor = last row of the previous page
   497‚Üí   497‚Üísupabase.from('items').select('id,name,merchant,amount,status,created_at')
   498‚Üí   498‚Üí  .eq('user_id', userId)
   499‚Üí   499‚Üí  .or(`created_at.lt.${cursor.createdAt},and(created_at.eq.${cursor.createdAt},id.lt.${cursor.id})`)
   500‚Üí   500‚Üí  .order('created_at', { ascending: false }).order('id', { ascending: false })
   501‚Üí   501‚Üí  .limit(20);
   502‚Üí   502‚Üí```
   503‚Üí   503‚Üí
   504‚Üí   504‚Üí**One round trip for the dashboard, not four:**
   505‚Üí   505‚Üí```sql
   506‚Üí   506‚Üícreate or replace function public.get_dashboard_summary()
   507‚Üí   507‚Üíreturns table (
   508‚Üí   508‚Üí  active_count bigint, expiring_soon_count bigint, expired_count bigint,
   509‚Üí   509‚Üí  subscription_monthly_total numeric, next_reminder_date date
   510‚Üí   510‚Üí)
   511‚Üí   511‚Üílanguage sql security invoker as $$
   512‚Üí   512‚Üí  select
   513‚Üí   513‚Üí    count(*) filter (where status = 'active'),
   514‚Üí   514‚Üí    count(*) filter (where status = 'expiring_soon'),
   515‚Üí   515‚Üí    count(*) filter (where status = 'expired'),
   516‚Üí   516‚Üí    coalesce(sum(amount) filter (where type = 'subscription' and billing_cycle = 'monthly' and status != 'cancelled'), 0),
   517‚Üí   517‚Üí    (select min(remind_on) from public.reminders r where r.user_id = auth.uid() and r.sent = false)
   518‚Üí   518‚Üí  from public.items where user_id = auth.uid();
   519‚Üí   519‚Üí$$;
   520‚Üí   520‚Üí```
   521‚Üí   521‚Üí`security invoker`, not `security definer` ‚Äî RLS still applies to the caller; the explicit `where` is defense-in-depth on top of it, not a replacement.
   522‚Üí   522‚Üí
   523‚Üí   523‚Üí**Avoid N+1 on item detail:** fetch an item and its reminders in one call via `items(*, reminders(*))` rather than a second round trip.
   524‚Üí   524‚Üí
   525‚Üí   525‚Üí**Before shipping any new query that touches `items` at scale, run `EXPLAIN ANALYZE` on it.**

   600‚Üí   600‚Üí## 16. Capacitor Packaging Path (Later ‚Äî Not Part of This Build)
   601‚Üí   601‚Üí
   602‚Üí   602‚Üí**Why this section exists, and why it's deliberately brief:** you don't need to touch this to ship the MVP. It exists so whoever builds the MVP (GLM 5.2, today) makes choices that don't box out this step later ‚Äî which is exactly what ¬ß3 already did. When you're actually ready:
   603‚Üí   603‚Üí
   604‚Üí   604‚Üí1. `npm install @capacitor/core @capacitor/cli @capacitor/android`
   605‚Üí   605‚Üí2. `npx cap init` ‚Äî `webDir: 'out'` (the static export output)
   606‚Üí   606‚Üí3. `next build` (produces `out/`) ‚Üí `npx cap add android` ‚Üí `npx cap sync`
   607‚Üí   607‚Üí4. Open in Android Studio, set the app icon from `icon-512.png` (already generated ‚Äî see the design doc), sign, build the APK.
   608‚Üí   608‚Üí5. Anything that assumed `window`/browser-only APIs should already be fine, since the whole app is already client-rendered per ¬ß3. Double-check any code that assumes a Vercel-hosted absolute URL (e.g., OAuth redirect URLs) ‚Äî those need to point at the Supabase-hosted Auth flow, not a Vercel domain, once there's no guaranteed web host in the loop.
   609‚Üí   609‚Üí
   610‚Üí   610‚ÜíThat's it ‚Äî this is a packaging step, not an architecture change, because the architecture already accounted for it.
